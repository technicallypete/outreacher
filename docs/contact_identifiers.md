# Contact Identifiers

## Problem

The original `app.leads` table used `email TEXT NOT NULL UNIQUE` as both the required field and the deduplication key. Gojiberry exports are LinkedIn-based and typically contain no email addresses, so every row was skipped during import.

## Solution

Deduplication is now handled by a separate `app.contact_identifiers` table. Email on `app.leads` is a nullable convenience column. Any identifier type — email, LinkedIn URL, Twitter handle, phone number, etc. — can serve as the unique key for a lead.

## Schema

```sql
CREATE TABLE app.contact_identifiers (
    lead_id  INTEGER NOT NULL REFERENCES app.leads(id) ON DELETE CASCADE,
    type     TEXT    NOT NULL,  -- 'email', 'linkedin', 'twitter', 'phone', ...
    value    TEXT    NOT NULL,
    PRIMARY KEY (type, value)
);
```

`(type, value)` is the primary key, so the same value cannot be assigned to two different leads regardless of type.

`app.leads.email` is now `TEXT` (nullable, no unique constraint). It is kept as a display/query convenience — the identifiers table is canonical for deduplication.

## Import logic

When importing a row the tool picks the primary identifier in priority order:

1. Email (if present)
2. LinkedIn Profile URL
3. Skip the row (no usable identifier)

**New lead:** inserted into `app.leads`, then all available identifiers for that row are written to `app.contact_identifiers`.

**Existing lead:** found via `FindLeadByIdentifier`, then fields are updated with `COALESCE` so existing values are not overwritten by nulls.

If a row has both email and LinkedIn URL, both are registered as identifiers so the lead can be found by either in the future.

## Migration

`mcp/migrations/003_contact_identifiers.sql`

- Creates `app.contact_identifiers`
- Migrates existing `email` and `linkedin_url` values from `app.leads` into the new table
- Drops the `NOT NULL` and `UNIQUE` constraints from `app.leads.email`
